## Архитектура тестов

### Технологический стек
- **Playwright**: Фреймворк для E2E тестирования
- **TypeScript**: Язык написания тестов
- **React**: Тестируемое приложение
- **Mantine UI**: Компоненты интерфейса

### Принципы тестирования
1. **Изоляция**: Тесты используют мокирование API для независимости от бэкенда
2. **Стабильность**: Добавлены retry механизмы и правильные ожидания
3. **Поддерживаемость**: Четкая структура файлов и вспомогательные функции
4. **Покрытие**: Тесты проверяют все ключевые пользовательские сценарии

## Установка и настройка

### Предварительные требования
- Node.js 18+
- npm или yarn
- Запущенные фронтенд и бэкенд серверы

### Установка зависимостей
```bash
cd frontend
npm install --save-dev @playwright/test
npx playwright install
```

### Конфигурация
Основные файлы конфигурации:
1. `frontend/playwright.config.ts` - основная конфигурация Playwright
2. `frontend/tests/helpers/test-utils.ts` - вспомогательные функции
3. `frontend/package.json` - скрипты для запуска тестов

## Запуск тестов

### Основные команды
```bash
# Запуск всех тестов в headless режиме
npm run test:e2e

# Запуск с UI для отладки
npm run test:e2e:ui

# Запуск в debug режиме
npm run test:e2e:debug

# Запуск в headed режиме (с видимым браузером)
npm run test:e2e:headed

# Запуск только в Chromium
npm run test:e2e:chromium
```

### Запуск с работающими серверами
```bash
# Терминал 1: Запуск бэкенда
cd backend && npm run dev

# Терминал 2: Запуск фронтенда
cd frontend && npm run dev

# Терминал 3: Запуск тестов
cd frontend && npm run test:e2e
```

## Тестовые сценарии

### 1. Базовая загрузка страницы
**Цель**: Проверить корректную загрузку страницы каталога
**Шаги**:
1. Открыть страницу `/book`
2. Проверить наличие заголовка "Выберите тип события"
3. Проверить наличие подзаголовка с инструкциями

**Ожидаемый результат**: Страница загружается, все элементы отображаются

### 2. Отображение карточек типов событий
**Цель**: Проверить отображение карточек с типами событий
**Шаги**:
1. Загрузить страницу
2. Проверить наличие минимум 3 карточек (fallback данные)
3. Проверить информацию на каждой карточке:
   - Длительность встречи
   - Название типа события
   - Иконка часов

**Ожидаемый результат**: Все карточки отображаются с корректной информацией

### 3. Навигация при клике на карточку
**Цель**: Проверить переход на страницу выбора слотов
**Шаги**:
1. Кликнуть на первую карточку
2. Проверить URL (должен содержать `/book/{eventTypeId}`)
3. Проверить загрузку страницы выбора слотов

**Ожидаемый результат**: Успешный переход на страницу выбора слотов

### 4. Состояние загрузки
**Цель**: Проверить отображение состояния загрузки
**Шаги**:
1. Замокать медленный ответ API (2 секунды задержки)
2. Перезагрузить страницу
3. Проверить отображение скелетонов
4. Дождаться загрузки и проверить исчезновение скелетонов

**Ожидаемый результат**: Скелетоны отображаются во время загрузки, исчезают после

### 5. Fallback при ошибке API
**Цель**: Проверить откат на локальные данные при недоступности API
**Шаги**:
1. Замокать ошибку API
2. Перезагрузить страницу
3. Проверить отображение fallback данных (3 карточки)

**Ожидаемый результат**: Отображаются fallback данные

### 6. Успешный запрос к API
**Цель**: Проверить отображение данных из API
**Шаги**:
1. Замокать успешный ответ API с кастомными данными
2. Перезагрузить страницу
3. Проверить отображение данных из API

**Ожидаемый результат**: Отображаются данные из API, а не fallback

## Структура файлов

```
frontend/
├── playwright.config.ts          # Конфигурация Playwright
├── tests/
│   ├── booking-catalog/
│   │   ├── booking-catalog.spec.ts      # Основные тесты
│   │   └── api-integration.spec.ts      # Тесты интеграции с API
│   └── helpers/
│       └── test-utils.ts                # Вспомогательные функции
└── package.json                  # Скрипты и зависимости
```

### Ключевые файлы

#### `playwright.config.ts`
Конфигурация Playwright с настройками для:
- Базового URL (`http://localhost:5173`)
- Поддержки нескольких браузеров (Chrome, Firefox, Safari)
- Генерации отчетов и скриншотов
- Настройки retry и timeout

#### `booking-catalog.spec.ts`
Основные тесты страницы каталога:
- Базовая загрузка
- Отображение карточек
- Навигация
- Состояние загрузки

#### `api-integration.spec.ts`
Тесты интеграции с API:
- Fallback при ошибке
- Успешный запрос к API

#### `test-utils.ts`
Вспомогательные функции:
- `mockEventTypesAPI()` - мокирование API ответа
- `mockAPIError()` - мокирование ошибки API
- `navigateToBookingCatalog()` - навигация на страницу каталога

## CI/CD интеграция

### GitHub Actions Workflow
Пример workflow для автоматического запуска тестов:

```yaml
name: E2E Tests
on: [push, pull_request]
jobs:
  e2e-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: cd frontend && npm ci
      - run: cd frontend && npx playwright install --with-deps
      - run: cd backend && npm ci
      - run: cd backend && npm run build
      - run: cd frontend && npm run build
      - run: |
          cd backend && npm start &
          cd frontend && npm run dev &
          sleep 10
          cd frontend && npm run test:e2e
```

### Локальная проверка перед коммитом
Рекомендуется добавить pre-commit hook:
```bash
# В package.json
{
  "scripts": {
    "precommit": "npm run test:e2e -- --grep \"@critical\""
  }
}
```

## Устранение неполадок

### Распространенные проблемы

#### 1. Тесты падают из-за timing issues
**Симптомы**: Элементы не находятся, хотя они есть на странице
**Решение**:
- Увеличить timeout: `await expect(element).toBeVisible({ timeout: 10000 })`
- Использовать `waitForLoadState`: `await page.waitForLoadState('networkidle')`
- Добавить явные ожидания: `await page.waitForTimeout(1000)`

#### 2. Селекторы не находят элементы
**Симптомы**: Ошибки "Element not found"
**Решение**:
- Проверить актуальные CSS классы в `BookingCatalogPage.tsx`
- Использовать data-testid атрибуты:
  ```typescript
  // В компоненте
  <div data-testid="event-type-card">
  
  // В тесте
  await page.getByTestId('event-type-card').click();
  ```
- Использовать role-based селекторы: `page.getByRole('heading', { name: '...' })`

#### 3. API моки не работают
**Симптомы**: Тесты используют реальный API вместо моков
**Решение**:
- Убедиться, что моки применяются до навигации:
  ```typescript
  test('test', async ({ page }) => {
    await page.route('**/event-types', handler); // Сначала мок
    await page.goto('/book'); // Потом навигация
  });
  ```
- Проверить URL в route: `**/event-types` должен совпадать с URL в коде

#### 4. Браузер не запускается
**Симптомы**: Ошибки "Browser not found"
**Решение**:
- Переустановить Playwright: `npx playwright install`
- Проверить системные зависимости: `npx playwright install-deps`
- Запустить в headed режиме для диагностики: `npm run test:e2e:headed`

### Отладка тестов

#### Playwright UI
```bash
npm run test:e2e:ui
```
Открывает интерактивный UI для пошагового выполнения тестов.

#### Playwright Debug
```bash
npm run test:e2e:debug
```
Запускает тесты в debug режиме с возможностью использования debugger.

#### Скриншоты и видео
При падении тестов автоматически создаются:
- Скриншоты последнего состояния
- Видео выполнения теста (если включено в конфигурации)
- Трассировка (trace) для анализа

## Рекомендации по поддержке

### 1. Обновление тестов при изменении UI
При изменении компонента `BookingCatalogPage.tsx`:
1. Обновить селекторы в тестах
2. Проверить новые пользовательские сценарии
3. Добавить тесты для новой функциональности

### 2. Добавление новых тестов
Для добавления нового теста:
1. Определить пользовательский сценарий
2. Создать тест в соответствующем spec файле
3. Использовать вспомогательные функции из `test-utils.ts`
4. Добавить моки при необходимости
5. Проверить работу теста локально

### 3. Оптимизация производительности
- Запускать тесты параллельно (уже настроено в конфигурации)
- Использовать кэширование зависимостей в CI
- Регулярно чистить отчеты и артефакты

### 4. Мониторинг стабильности
- Отслеживать flaky тесты (нестабильные тесты)
- Использовать retry механизм для временных проблем
- Регулярно обновлять Playwright до последней версии

## Заключение

Созданные E2E тесты обеспечивают:
- **Полное покрытие** ключевых пользовательских сценариев
- **Стабильность** благодаря изоляции через мокирование
- **Поддерживаемость** через четкую структуру и документацию
- **Интеграцию** с CI/CD для автоматической проверки

Тесты готовы к использованию и могут быть расширены для покрытия других страниц приложения.
